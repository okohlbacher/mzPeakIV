<objective>
Create src/worker/mzPeakWorker.ts — the Worker entry point that relocates the entire runLoad pipeline and on-demand render calls from the main thread into a Worker thread. The Worker holds the live Reader handle and all compute calls; the main thread is left stateless with respect to file I/O.

Purpose: This is the core deliverable of Phase 5. Moving all blocking I/O and compute off the main thread is what makes the app responsive on real-scale files (Phase 5 success criterion 1). It also isolates malformed-file crashes to the Worker thread so the UI stays alive.

Output: src/worker/mzPeakWorker.ts — a single self-contained Worker file with the complete load and render pipeline, typed against protocol.ts from Plan 01.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Main thread → Worker | postMessage WorkerRequest — only strings, numbers, ArrayBuffers |
| Worker (file I/O) | Malformed .mzpeak bytes fed to WASM parser — Worker isolates crash to its thread |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02 | Tampering | renderIonImage input validation | mitigate | V5 guard replicated inside Worker: !isFinite(mz) || mz <= 0 || ... checks run at processing site, not just at dispatch |
| T-05-03 | Tampering | Malformed file bytes in WASM | mitigate | Worker isolates WASM crash; postError catches thrown Error/UnsupportedEncodingError and serializes to plain object; main thread stays alive |
| T-05-04 | Tampering | Float32Array view transferred instead of buffer | mitigate | Always [float32Array.buffer] in transfer list; view set to null after postMessage |
| T-05-SC | Tampering | npm/pip/cargo installs | accept | No new packages; no install tasks in this plan |
</threat_model>
